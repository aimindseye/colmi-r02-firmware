## Container structure

- header size: `0x50`;
- magic: `e5 c3 bd 81`;
- payload length at `0x04` and `0x08`;
- byte-sum32 at `0x0C`;

### Older R02 `.06` and `.17`

- header size: `0x100`;
- CRC32 integrity;
- different binary lineage and layout.

## Family comparison

`RY02_3.00.33_250117.bin` is the only direct predecessor found in the ATC corpus. The low same-offset similarity is caused by relocation/linker layout changes, while large numbers of exact blocks and matched function movement establish common lineage.

Important relocations:

```text
A1 handler:     0x1CE2 -> 0x1D02  (+0x20)
raw timer site: 0x1DBE -> 0x1DDE  (+0x20)
timer wrapper:  0x36B4 -> 0x376C
```

## Raw command path

The A1 command handler selects subcommands and configures the raw-stream timer.

At `.38` payload offset `0x1DDE`:

```asm
ldr   r0, =0x002098C4
movs  r2, #0x7d
movs  r3, #1
lsls  r2, r2, #3      ; 1000
ldr   r1, =0x00825B45
adds  r0, #0x10       ; timer object 0x002098D4
bl    0x376c
```

There are no writes to `r2` between the `lsls` and the call.

## Timer callback

Callback pointer mapping:

```text
0x00825B45 & ~1 = 0x00825B44
0x00825B44 - 0x00824000 = payload offset 0x1B44
```

The callback at `0x1B44` sends three raw packet types every invocation:

- A101 packet through send function `0x66C0`;
- A102 packet through send function `0x66C0`;
- A103 packet through send function `0x66C0`.

No independent one-second gating logic was found before these sends. Later counters are termination/watchdog behavior rather than per-packet rate limiting.

## Timer wrapper

The `.38` wrapper at `0x376C`:

- passes `r2` unchanged when updating an existing timer;
- passes the original `r2` as the period when creating a timer;
- performs no clamp, division, rounding, or minimum-period comparison.

The wrapper is instruction-equivalent to `.33` aside from relocated local branch targets.

## Scheduler capability

Other callers use the same timer wrapper with periodic values below 1000 milliseconds:

```text
10, 30, 40, 100, 320, 400 ms
```

This proves the platform scheduler accepts a periodic `256 ms` value.

## Patch mechanics

Experimental change:

```text
stock:   125 << 3 = 1000 ms
patched:  32 << 3 =  256 ms
```

Container differences:

```text
header offset  0x00000C: 0xFC -> 0x9F
payload offset 0x001E2E: 0x7D -> 0x20
```

Only two bytes differ: the timer immediate and the required sum32 byte. Firmware and hardware strings are unchanged. Length and integrity checks pass.

The older public ATC `.06` modification follows the same pattern: one payload byte plus four CRC32 bytes.

## Runtime result

The transfer reached all command-3 block acknowledgements, command 4 returned success, and command 5 was sent. After reboot, raw A101/A102/A103 remained at exactly `1.00 Hz`.

An executing `256 ms` timer should produce approximately:

```text
1000 / 256 = 3.90625 callback cycles per second
```

Therefore the modified application was not the code running after reboot.

## Official QRing DFU flow

BLE OTA UUIDs:

```text
service: de5bf728-d711-4e47-af26-65e3012a5dc7
notify:  de5bf729-d711-4e47-af26-65e3012a5dc7
write:   de5bf72a-d711-4e47-af26-65e3012a5dc7
```

Frame format:

```text
0       0xBC
1       command
2..3    payload length
4..5    CRC16(payload)
6..     payload
```

State machine:

```text
ACK 1 -> send command 2 metadata
ACK 2 -> begin command 3 blocks
ACK 3 -> next 1024-byte block; after final block report 100%
ACK 3 after data complete -> activity sends command 4
ACK 4 -> activity calls endAndRelease(), sends command 5
```

The official app intentionally clears callbacks immediately after queueing command 5. It does not wait for command-5 acknowledgement. No command 6 or separate BLE activation step was found.

## Remaining hypotheses

1. boot-time internal checksum/digest/signature validation;
2. same-version activation suppression;
3. trial boot followed by rollback because confirmation is missing;
4. browser command-5 write race, although this is weaker than the boot-validation hypotheses.
